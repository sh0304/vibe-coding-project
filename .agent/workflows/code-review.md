# 아키텍처 및 코드 정밀 리뷰 워크플로우

본 워크플로우는 코드가 단순히 '동착하는가'를 넘어, **프로젝트의 지속 가능성과 설계의 건전성**을 4가지 핵심 지표를 기준으로 정밀 진단합니다.

## 1. 진단 지표 및 체크리스트

### 지표 1: 도메인 기반 아키텍처 (Structure & Domain)
- [ ] 관련 파일들이 `src/features/[도메인명]` 하위에 응집도 있게 모여 있는가?
- [ ] 도메인 간의 의존성이 꼬여 있지 않고, Server Actions가 해당 피처에 귀속되어 있는가?
- [ ] RSC(Server)와 Client 영역이 `use client` 지시어를 통해 명확하고 효율적으로 분리되었는가?

### 지표 2: 코드 공통화 및 추상화 (Commonization)
- [ ] 중복으로 사용되는 UI 요소가 `src/components/ui` 또는 `shared/components`로 자산화되었는가?
- [ ] 여러 액션에서 반복되는 DB 쿼리나 유틸리티 로직이 `src/lib` 등으로 공통화되어 있는가?
- [ ] Props Drilling이 과도하지 않은지, 데이터 흐름이 단순하고 명확하게 설계되었는가?

### 지표 3: 코드 청결도 및 정돈 (Code Hygiene)
- [ ] 사용되지 않는 변수, 임포트, 죽은 코드(Dead Code)가 방치되어 있지는 않은가?
- [ ] 복잡한 if-else 패턴이나 중첩된 로직이 가독성을 해치고 있지 않은가?
- [ ] 변수 및 함수명이 비즈니스 의도를 명확히 드러내며 일관된 명명 규칙을 따르는가?

### 지표 4: 성능 최적화 및 리팩토링 (Optimization)
- [ ] Prisma 쿼리 시 불필요한 필드 조회나 N+1 문제를 유발하는 루프가 포함되지 않았는가?
- [ ] 무거운 클라이언트 상태 관리가 남용되지 않고, 필요한 곳에만 `useState`/`Context`가 쓰였는가?
- [ ] SCD Type 2 로직이 `$transaction`을 통해 원자성을 보장하며 최적의 속도로 수행되는가?

## 2. 리뷰 및 리팩토링 실행 프로세스

> **대전제**: 모든 리팩토링 및 최적화 작업은 **기존 기능이 이전과 100% 동일하게 정상 동작함**을 전제로 합니다. 기능적 변경(Breaking Changes)이 발생해서는 안 됩니다.

1. **기능 스냅샷 및 정적 분석**: 수정 전 현재 기능의 정상 동작 상태를 파악하고, 위 4대 지표에 따라 리팩토링 대상을 선정합니다.
2. **개선안 도출 및 교정**: 구조적 최적화가 필요한 경우 구체적인 리팩토링 가이드와 `diff` 코드를 작성하여 적용합니다.
3. **무결성 검증 (Regression Test)**: 
   - 수정 후 `testing` 스킬을 활용하여 기존 단위 테스트가 모두 통과하는지 확인합니다.
   - `browser-qa` 워크플로우를 실행하여 사용자 시나리오상 오작동이 없는지 최종 컨펌합니다.
   - 기능이 하나라도 어긋날 경우, 즉시 롤백하거나 수정 전 상태로 기능을 복구합니다.
